# Code Analysis Skill

## 我能做什么

- **代码探索** - 理解项目结构、技术栈
- **代码审计** - 发现代码问题、优化点
- **架构分析** - 分析系统架构、设计模式
- **依赖分析** - 分析项目依赖、模块关系

## 何时使用

当你需要：
- 理解一个新项目的代码结构
- 审计现有代码，发现问题
- 提出代码优化方案
- 分析技术选型和架构
- 做项目尽调

## 分析维度

### 1. 项目结构

```
项目根目录/
├── src/                    # 源代码
├── tests/                  # 测试
├── docs/                   # 文档
├── config/                 # 配置
└── ...
```

检查：
- 目录结构是否清晰
- 模块划分是否合理
- 文件组织是否符合规范

### 2. 技术栈

分析使用的：
- 编程语言及版本
- 框架和库
- 数据库
- 构建工具
- 包管理器

### 3. 代码质量

检查：
- 代码风格一致性
- 命名规范
- 注释覆盖率
- 函数/类大小
- 重复代码
- 安全问题

### 4. 架构设计

分析：
- 设计模式使用
- 模块间依赖
- API 设计
- 数据流

### 5. 测试覆盖

检查：
- 测试文件存在性
- 测试覆盖率
- 测试类型（单元/集成/E2E）

## 分析步骤

### 第一步：探索项目结构

```bash
# 查看目录结构
ls -la

# 查看文件类型分布
find . -type f | head -50

# 查看配置文件
ls -la *.json *.yaml *.toml 2>/dev/null
```

### 第二步：分析技术栈

```bash
# 前端
cat package.json  # Node.js
cat requirements.txt  # Python
cat go.mod  # Go

# 后端
cat pom.xml  # Java
cat Gemfile  # Ruby
```

### 第三步：代码质量分析

```bash
# 统计代码行数
find . -name "*.py" -exec wc -l {} \; | sort -n

# 检查大文件
find . -type f -size +1000k

# 查找潜在问题
grep -r "TODO\|FIXME\|BUG" --include="*.py"
```

### 第四步：生成报告

整理发现的问题和改进建议。

## 检查清单

- [ ] 有测试覆盖
- [ ] 敏感信息未硬编码
- [ ] 无明显安全漏洞

## 常见问题模式

### 代码重复
- 相同逻辑出现 3+ 次
- 建议：抽取为公共函数

### 函数过长
- 函数超过 100 行
- 建议：拆分职责

### 紧耦合
- 模块间直接引用过多
- 建议：引入抽象层

### 魔法数字
- 代码中出现未解释的数字
- 建议：定义为常量

### 缺乏测试
- 核心逻辑无测试
- 建议：补充测试

## 优化建议优先级

| 优先级 | 类型 | 原因 |
|--------|------|------|
| P0 | 安全漏洞 | 可能造成数据泄露或系统崩溃 |
| P1 | 严重性能问题 | 影响用户体验 |
| P2 | 代码异味 | 维护成本高，容易引入 bug |
| P3 | 代码风格 | 统一团队规范 |
| P4 | 优化建议 | 提升可读性 |

## 注意事项

- 保持客观，基于事实
- 提供具体改进建议，而非只提问题
- 考虑项目实际约束（时间、人力）
- 区分核心问题和风格问题
